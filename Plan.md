# ПЛАН ТЕСТИРОВАНИЯ МОБИЛЬНОГО ПРИЛОЖЕНИЯ

## Исполнитель: Кащеева Наталья Олеговна

## 1. Что тестируем?
### 1.1. Обзор приложения
- **Назначение:** Мобильное приложение для работы с новостями хосписа
- **Основные модули:**
    - Авторизация пользователей
    - Работа с новостями (создание, просмотр, редактирование, фильтрация)
    - Просмотр тематических цитат
    - Информация о приложении
- **Целевая платформа:** Android API 29+ (Android 11+)

- **Основная активити запуска:**
   android:name="ru.iteco.fmhandroid.ui.AppActivity".
	
- **Основной класс приложения:**
   android:name="ru.iteco.fmhandroid.application.FhmApplication".

### 1.2. Границы тестирования
**Включается:**
- Функциональное тестирование основных сценариев использования
- UI-тестирование всех экранов приложения
- Тестирование авторизации и навигации
- Проверка работы с новостями
- Smoke и регрессионное тестирование

**Исключается:**
- Тестирование на версиях Android ниже 11
- Performance-тестирование под нагрузкой
- Security-аудит и пентестинг
- Кросс-платформенное тестирование

## 2. Как будем тестировать?
### 2.1. Типы тестирования
- **Функциональное тестирование:** Проверка работоспособности функций
- **UI-тестирование:** Проверка интерфейса и пользовательского опыта
- **Smoke-тестирование:** Быстрая проверка стабильности сборки
- **Регрессионное тестирование:** Проверка после изменений кода

### 2.2. Техники тест-дизайна
- **Эквивалентное разделение:** Группировка входных данных
- **Анализ граничных значений:** Проверка граничных условий
- **Таблицы принятия решений:** Для сложной бизнес-логики
- **Тестирование состояний и переходов:** Для навигации
- **Сценарии использования (Use Cases):** Основные пользовательские сценарии

### 2.3. Стратегия автоматизации
- **Инструменты:** Espresso, JUnit, Allure, Android Studio
- **Подход:** Page Object Pattern + Step-by-Step
- **Архитектура:** Модульное разделение тестов по функциональности
- **Отчетность:** Allure отчеты с аннотациями и скриншотами

## 3. Когда будем тестировать?
### 3.1. Критерии начала тестирования
1. **Готовность тестового окружения:**
    - Установлен и настроен Android Studio
    - Развернут эмулятор Android API 29+
    - Настроен CI/CD (опционально)
2. **Готовность приложения:**
    - Собранная APK версия 1.0.0
    - Доступ к тестовому серверу
    - Тестовые данные подготовлены
3. **Готовность документации:**
    - Утвержден тест-план
    - Составлены чек-лист и тест-кейсы
    - Определены роли и ответственность

### 3.2. Этапы тестирования
1. **Подготовительный этап:** 1 день
2. **Ручное тестирование:** 3 дня
3. **Автоматизация тестов:** 5 дней
4. **Регрессионное тестирование:** 1 день
5. **Формирование отчетов:** 1 день

### 3.3. Критерии окончания тестирования
1. **Критерии полноты:**
    - Выполнены все тест-кейсы высокого приоритета
    - Автоматизированы основные пользовательские сценарии
    - Критические и блокирующие баги исправлены
2. **Критерии качества:**
    - Показатель успешных тестов ≥ 90%
    - Все smoke-тесты проходят успешно
    - Отчеты Allure сформированы и доступны
3. **Критерии времени:**
    - Выполнены все запланированные этапы
    - Подготовлена итоговая документация

## 6. Возможные риски и пути их разрешения
### 6.1. Риски при ручном тестировании
|           Риски            |  Вероятность  |  Влияние  |               Меры предотвращения                |
|:--------------------------:|:-------------:|:---------:|:------------------------------------------------:|
| Недостаток тестовых данных |    Средняя    |  Высокое  |        Подготовка тестовых данных заранее        |

### 6.2. Риски при автоматизации
|       Риски        |  Вероятность  |  Влияние  |          Меры предотвращения           |
|:------------------:|:-------------:|:---------:|:--------------------------------------:|

1. **Регулярные code review** тестового кода

3. **Резервное копирование** тестовых данных
4. **Мониторинг стабильности** тестов в CI/CD

## 8. Сортировка тест-кейсов по автоматизации
### 8.1. Автоматизировать (высокий приоритет):
- Авторизация (успешная/неуспешная)
- Навигация между разделами
- Создание и просмотр новостей
- Проверка основных экранов (главный, цитаты, о приложении)

### 8.2. Автоматизировать (средний приоритет):
- Фильтрация новостей
- Редактирование новостей
- Обновление данных (pull-to-refresh)
- Проверка валидации полей

### 8.3. Проверять вручную (низкий приоритет):
- Поведение при смене ориентации
- Работа при потере сети
- Внешние ссылки (открытие в браузере)
- Визуальная проверка UI/UX


## 9. Метрики успешности
- **Покрытие тестами:** ≥ 70% основных сценариев
- **Успешность тестов:** ≥ 90% прохождения
- **Время выполнения:** Все тесты ≤ 10 минут
- **Стабильность:** Флаки тесты ≤ 5%
